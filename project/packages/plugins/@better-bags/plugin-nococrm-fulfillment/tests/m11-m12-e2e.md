# M11 + M12 端到端测试清单 (T11.08 + T12.07)

> 在 NocoBase UI 完成"手动配置清单"后执行。每条 30 秒至 2 分钟可验证。

## 准备
- 启用 plugin-nococrm-core + plugin-nococrm-quotation + plugin-nococrm-orders + plugin-nococrm-fulfillment
- 启用 plugin-kanban / plugin-workflow* / plugin-notification-email / plugin-field-sequence
- M9 已有至少 1 个 status=production_pending 或更后状态的 order（含 production_plan 记录）
- 该 order 关联了 contract / quotation / customer

---

## 场景 1：生产进度累加（T11.01）
1. 给某个 production_plan 创建第 1 条 production_progress：date=今天-2，completed_qty=100
2. 创建第 2 条：date=今天-1，completed_qty=150
3. 创建第 3 条：date=今天，completed_qty=200
- 期望：
  - [ ] 第 1 条 cumulative_qty = 100
  - [ ] 第 2 条 cumulative_qty = 250
  - [ ] 第 3 条 cumulative_qty = 450
  - [ ] order 字段自动 = production_plan.order_id

## 场景 2：生产异常通知业务员（T11.04）
1. 创建 production_progress：issues="某批次面料色差需要补料"
2. 保存
- 期望：
  - [ ] 关联 order.notes 自动追加 `[YYYY-MM-DD PROD_ISSUE] 某批次面料色差需要补料`

## 场景 3：QC 报告 + AQL 默认（T11.02 + T11.03）
1. 给某个 order 创建 qc_report
2. 不填 aql_level
- 期望：
  - [ ] aql_level 默认 = 2.5
  - [ ] report_no 自动 `QC-YYYYMMDD-001`
  - [ ] type 默认 = aql_internal

## 场景 4：QC 通过不是最终内检不推进订单
1. 创建 qc_report：pass=true，is_final_internal=false
2. 保存
- 期望：
  - [ ] order.status 不变（保持 in_production 或 qc_pending）

## 场景 5：QC 最终通过推进订单（T11.06 核心）
1. 创建 qc_report：pass=true，is_final_internal=true
2. 保存
- 期望：
  - [ ] order.status 自动 = `completed`
  - [ ] 邮件由 README 定时工作流触发（手动验证或 mock）

## 场景 6：QC 通过但订单已是更后状态（不回滚）
1. 给一个 status=shipped 的 order 创建 qc_report：pass=true，is_final_internal=true
- 期望：
  - [ ] order.status 保持 shipped（hook 检测到 currentIdx >= completedIdx，不回滚）

## 场景 7：客户验厂字段就位（T11.05）
1. 打开任意 order
- 期望：
  - [ ] 看到 `customer_inspection_required` 复选框字段
  - [ ] 默认 = false

## 场景 8：发货创建尾款校验（T12.02）
1. 找一个 order：customer.status=new（新客户），balance_paid=false
2. 尝试给它创建 shipment
- 期望：
  - [ ] 报错"未收到尾款，不能创建 shipment..."

## 场景 9：尾款已付可以发货（T12.02）
1. 把 order.balance_paid 改为 true（模拟尾款到账，正常由 M9 payments hook 自动维护）
2. 创建 shipment
- 期望：
  - [ ] 创建成功
  - [ ] order.status 自动 = `shipping_pending`
  - [ ] shipment_no 自动 `SH-YYYYMMDD-001`

## 场景 10：VIP 例外允许未付尾款发货（T12.02）
1. 找一个 VIP 客户（status=closed AND 3+ orders）
2. 该客户某 order 的 balance_paid=false
3. 创建 shipment
- 期望：
  - [ ] 创建成功（VIP 例外放行）

## 场景 11：提单号录入自动填 ATD（T12.03）
1. 给场景 9 的 shipment 录入 bl_no="ZTPC8848777"
2. 保存
- 期望：
  - [ ] atd 自动 = 当前时间
- 1. 把 atd 改为 5 天前，再改 bl_no
- 期望：
  - [ ] atd 保留 5 天前（hook 仅在 atd 为空时填，不覆盖）

## 场景 12：文件齐套校验（T12.05）
1. 找一个 shipment，没有上传 invoice_file / packing_list_file / origin_cert_file

- 1. 上传三个文件
2. 再次切 status=shipped
- 期望：
  - [ ] 保存成功

## 场景 13：发货后订单状态同步（T12.04）

- 期望：
  - [ ] order.status 自动 = `delivered`

## 场景 14：状态推送邮件触发（T12.04）

2. shipment.status 变更
- 期望：

## 已知缺陷与未覆盖项
- 验厂任务 / 尾款邀请邮件 / 提单号通知 / 状态推送 4 个工作流依赖 plugin-workflow-schedule
- 字段级 ACL（payments 财务隔离）留 M15
- cumulative_qty 假设同一 plan 内 date 严格递增；同一天多条进度可能产生小误差（按 id 排序兜底）
- delivered_at 由 hook 在 status=delivered 时自动填，但若业务员先填 delivered_at 再切 status 顺序倒置 hook 不强制
