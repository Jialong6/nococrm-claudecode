# M9 + M10 端到端测试清单 (T9.08 + T10.07)

> 在 NocoBase UI 完成"手动配置清单"后执行。每条 30 秒至 2 分钟可验证。

## 准备
- 启用 `plugin-nococrm-core` + `plugin-nococrm-quotation` + `plugin-nococrm-orders`
- 启用 `plugin-kanban` / `plugin-workflow*` / `plugin-gantt` / `plugin-field-sequence` / `plugin-field-formula`
- M5/M6 已有至少 1 个 status='confirmed' 的 quotation（关联 customer + inquiry）
- M5/M6 的 quotation 有以下字段填好：quantity / unit_price / production_factory（缅甸或山东都行）/ incoterms / payment_terms / lead_time_days

---

## 场景 1：合同创建自动同步报价信息（T9.01）
1. 创建 contract，选 quotation = 现成的某个报价
2. 不填 customer / total_amount / currency / incoterms / payment_terms
3. 保存
- 期望：
  - [ ] contract_no 自动 `CT-YYYYMMDD-001`
  - [ ] customer 自动 = quotation.customer
  - [ ] total_amount 自动 = quotation.total（unit_price × quantity）
  - [ ] currency / incoterms / payment_terms 自动同步

## 场景 2：合同签署自动建单 + 定金（T9.04 核心）
1. 把场景 1 的 contract.status 从 pending 改为 signed
2. 填 signed_at = 今天
3. 保存
- 期望：
  - [ ] 数据源里 orders 表自动新建一条记录
    - order_no 自动 `PO-YYYYMMDD-001`
    - contract_id = contract.id
    - customer_id = contract.customer_id
    - quantity / unit_price / production_factory_id 同步自 quotation
    - status = `deposit_pending`
    - deposit_paid = false, balance_paid = false
  - [ ] payments 表新建一条定金记录
    - type = `deposit`
    - amount = total_amount × 30%
    - percentage = 30
    - due_date = signed_at + 7 天
    - status = `unpaid`
  - [ ] 关联的 customer.status 改为 `ordered`

## 场景 3：合同重复保存的幂等性（T9.04）
1. 再次保存场景 2 的 contract（不改任何字段）
- 期望：
  - [ ] orders 不会重复创建（仍只有 1 条关联记录）
  - [ ] payments 不会重复创建

## 场景 4：缅甸订单 MOQ < 1500 阻断（T9.02 业务红线）
1. 创建一个 quotation，production_factory = MM_YGN，quantity = 800
2. 创建合同 → signed
3. orders 自动创建后，手动把 quantity 改成 800（< 1500）
4. 保存
- 期望：
  - [ ] 报错"缅甸大货 MOQ 不得低于 1500 件..."
  - [ ] 保存被阻断

## 场景 5：缅甸订单交期 < PP + 90 天阻断（T9.02）
1. 用一个缅甸 order（quantity ≥ 1500）
2. 填 pp_sample_approved_at = 今天
3. 填 committed_delivery = 60 天后
4. 保存
- 期望：
  - [ ] 报错"缅甸大货承诺交期不得早于 PP 样确认日 + 90 天..."

## 场景 6：定金到账触发排产 + 尾款（T9.05 核心）
1. 找到场景 2 的定金 payment
2. 把 status 从 unpaid 改为 paid，填 paid_at = 今天
3. 保存
- 期望：
  - [ ] 关联的 order.status 变成 `production_pending`
  - [ ] order.deposit_paid = true
  - [ ] payments 表新建一条尾款记录
    - type = `balance`
    - amount = total × 70%
    - percentage = 70
    - due_date = committed_delivery - 7（若 committed_delivery 已填）
  - [ ] production_plans 表新建一条记录
    - order_id 关联此 order
    - factory_id = order.production_factory_id
    - status = `pending`
    - special_requirements = order.notes（若有）

## 场景 7：尾款到账（T9.05）
1. 把场景 6 的尾款 payment.status 改为 paid
2. 保存
- 期望：
  - [ ] order.balance_paid = true
  - [ ] order.status 保持当前值（status 推进留 M12 发货流程）

## 场景 8：materials 时间轴自动计算（T10.03）
1. 给某个 order 创建一条 material：item_name="尼龙面料", po_date=今天
2. 保存
- 期望：
  - [ ] expected_dongguan_arrival 自动 = 今天 + 14 天
  - [ ] expected_qc_pass 自动 = 今天 + 21 天
  - [ ] expected_factory_arrival 自动 = 今天 + 35 天

## 场景 10：production_plans 5 里程碑（T10.02）
1. 找到场景 6 自动创建的 production_plan
2. 填 planned_start / planned_end / 5 个里程碑日期

- 期望：

## 场景 11：物料逾期定时工作流（T10.04）
1. 按 README 配置定时（每日 9:00）
2. 找一条 material，设 supplier_eta_dg = 昨天，actual_dg_arrival 留空
3. 手动触发或等定时
- 期望：
  - [ ] 采购部收到任务"物料 X 逾期未到东莞"

## 场景 12：排产 -3d 通知质检（T10.05）
1. 按 README 配置定时
2. 给某个 production_plan 设 planned_start = 3 天后，status = pending
3. 手动触发
- 期望：
  - [ ] QC 团队收到任务
  - [ ] 任务内容含 special_requirements

## 场景 13：M5/M6 hook 现在完整生效（连锁验证）
1. 此插件启用前，M5/M6 的 T6.03（新客户禁选 tt_bl）依赖 orders.count，orders 表不存在时静默返回 0
2. 现在 orders 表存在了
3. 找一个 customer.status = closed 的客户（VIP），并已有 3+ 个 orders
4. 给这个客户的某个 quotation 设 payment_terms = tt_bl
- 期望：
  - [ ] 不报错（orders.count >= 3 + customer.status = closed → 通过 VIP 校验）
- 1. 找一个 customer.status = new 的客户（新客户，0 orders）
2. 同上设 payment_terms = tt_bl
- 期望：
  - [ ] 报错"T/T 见提单复印件付款 仅 VIP..."（现在 orders.count = 0 真实反映）

## 已知缺陷与未覆盖项
- 财务字段级 ACL 完整实现留 M15
- materials formula 字段在不同 SQL 方言中表现不一致（MySQL DATE_ADD vs PostgreSQL date 算术）；如出错可在 UI 改成 plugin-field-formula 的 dayjs 表达式
- production_plans 的 milestone 字段不强制顺序检查（业务员可乱序填）；后续 M11 引入生产进度记录时再约束
